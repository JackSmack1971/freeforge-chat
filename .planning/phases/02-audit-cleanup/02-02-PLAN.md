<objective>
Keep the Content Security Policy locked down at the delivery layer by preserving the `base-uri`
directive in `netlify.toml`.

Purpose: the v1.0 audit flagged the lack of `base-uri` as defense-in-depth debt. FreeForge already
ships its CSP as an HTTP header, so this phase keeps the header as the single source of truth and
ensures `base-uri 'none'` remains present in that header string.

Output: `netlify.toml` continues to publish the hardened CSP header with `base-uri 'none'`.
</objective>

## Phase Goal

**As a** browser loading FreeForge, **I want to** receive a CSP that also locks down `<base>` tag
injection, **so that** relative URLs cannot be retargeted to an attacker-controlled origin.

<tasks>

<task type="auto">
  <name>Task 1: Keep base-uri in the Netlify CSP header</name>
  <files>netlify.toml</files>
  <read_first>
    - netlify.toml — read the existing `[[headers]]` block and the `Content-Security-Policy` value
  </read_first>
  <action>
    Ensure the authoritative `Content-Security-Policy` header includes `base-uri 'none'` or
    `base-uri 'self'` alongside the existing directives.

    Preserve the current header-driven CSP approach:
    - do not introduce a duplicate CSP meta tag in `freeforge/index.html`
    - do not relax any existing directive
    - keep the rest of the header string unchanged unless the base-uri directive needs to be added
  </action>
  <verify>
    <automated>rg -n "Content-Security-Policy|base-uri 'none'|base-uri 'self'" netlify.toml</automated>
    <automated>rg -n "Content-Security-Policy" freeforge/index.html</automated>
  </verify>
  <acceptance_criteria>

  </acceptance_criteria>
  <done>The Netlify-delivered CSP still enforces `base-uri` and keeps the defense-in-depth hardening intact</done>
