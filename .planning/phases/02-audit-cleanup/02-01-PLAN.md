<objective>
Mirror the onboarding empty-models guard in `freeforge/src/features/settings.js` so a key that
authenticates but returns no free models is treated as unusable instead of being saved as success.

Purpose: this closes the remaining review warning from the v1.0 audit without broadening the phase.
The fix should preserve the existing success flow for usable keys and only block the false-positive
case that onboarding already handles.

Output: `freeforge/src/features/settings.js` rejects zero-model results with a visible error and
does not persist the key or refresh the chat state on that path.
</objective>

## Phase Goal

**As a** user updating an API key in settings, **I want to** be told when the key has no usable free
models, **so that** I do not save a dead key and end up with an unusable account state.

<tasks>

<task type="auto">
  <name>Task 1: Add the empty-models guard to settings.updateKey()</name>
  <files>freeforge/src/features/settings.js</files>
  <read_first>
    - freeforge/src/features/settings.js — read the full file, then locate `updateKey()` and the current `fetchFreeModels(key)` success path
  </read_first>
  <action>
    Insert the onboarding-style guard immediately after `const models = await fetchFreeModels(key);`.

    If `models.length` is zero:
    - set `#settings-key-error` text to `No free models found for this key`
    - unhide the error element
    - return before `S.apiKey = key`, `S.models = models`, `setStoredKey(key)`, `hideInvalidBanner()`,
      `populateModelsFromState()`, `closeSettings()`, or the success toast

    Keep the existing success path unchanged for usable keys.
  </action>
  <verify>
    <automated>rg -n "No free models found for this key|settings-key-error|fetchFreeModels\\(key\\)" freeforge/src/features/settings.js</automated>
    <automated>rg -n "S\\.apiKey = key|setStoredKey\\(key\\)|populateModelsFromState\\(\\)|closeSettings\\(\\)" freeforge/src/features/settings.js</automated>
  </verify>
  <acceptance_criteria>
    - `updateKey()` has a guard that returns when `models.length === 0`
    - The guard writes `No free models found for this key` into `#settings-key-error`
    - The guard runs before any persisted state update or success toast
    - Usable keys still follow the current success path unchanged
  </acceptance_criteria>
  <done>`settings.js` matches onboarding behavior for zero-model API keys and no longer saves a dead key as success</done>
