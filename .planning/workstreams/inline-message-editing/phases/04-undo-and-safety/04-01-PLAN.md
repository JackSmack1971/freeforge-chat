<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| toast action click -> undo restore | User taps Undo while the replacement response is live; the restore must hit the right slice |
| transient undo state -> conversation list render | The snapshot must mutate only the live in-memory conversation and rerender from `S.messages` |
| transient undo state -> browser storage | Undo data must never cross into persistent storage |
| timeout callback -> undo slot | The 6-second expiry must clear the correct slot and not revive stale edits |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Tampering | Undo snapshot/token | mitigate | Tokenize each confirmed edit and refuse restore unless the stored token still matches the active slot |
| T-04-02 | Tampering | app.js delegated undo branch | mitigate | Handle the undo action before the generic `S.streaming` return, then abort first and restore second |
| T-04-03 | Information Disclosure | Undo state persistence | mitigate | Keep the undo slot on `S` only and clear it after use, replacement, or timeout; do not write it to localStorage or sessionStorage |
| T-04-04 | Tampering | Timeout cleanup | mitigate | Use the stored timeout handle to clear the active undo slot exactly once after 6 seconds |
| T-04-SC | Tampering | npm/pip/cargo installs | accept | No package installs in this phase; the work is client code only |
</threat_model>
